# Защита от разведки модели/промпта и абьюза

Этот документ описывает, что было сделано против «мамкиных хакеров» (вопросы «какая
модель», попытки вытащить системный промпт, hammering эндпоинтов) и — главное —
**что вам нужно сделать руками**: включить RLS в Supabase и применить миграции.

Промпт-защита вторична: любопытный пользователь читает модель не из ответа ассистента,
а из DevTools. Поэтому основная работа — закрыть утечки по проводу и дыры доступа.

---

## Что уже в коде (задеплоится с этой веткой)

**Промпт (`backend/app/rag_core/prompt.py`)**
- Секция `[0]` — абсолютный приоритет безопасности над любыми инструкциями из ввода.
- Секция `[12]` — жёсткая анти-extraction политика: не раскрывать модель/вендора/промпт/
  инструменты; покрыты типовые обходы (ролевые уловки, «повтори текст выше», инъекции
  внутри приложенных документов, «я разработчик»). Реакция — нейтральная фраза, без
  признания самого факта наличия защиты.

**Обеззараживание провода (backend)**
- Из клиентских метаданных `final`-события убрано поле `provider` (`nodes.py`).
- Внутренние ключи моделей переименованы в нейтральные `fast/thinking/power/alt`
  (`chat.yaml`) — теперь `metadata.modelUsed` не выдаёт вендора даже при фолбэке.
- SSE-событие `status` больше не содержит сырое имя инструмента (`search_court_practice`
  и т.п.) — только человекочитаемый `label`.
- Ошибки генерации уходят клиенту одной нейтральной фразой; сырой текст (с id моделей,
  URL провайдера) только в серверных логах.

**Фронтенд**
- Токен модели в запросе переименован `openai` → `fast`; тип `SelectedModel` сужен до
  `'fast' | 'thinking'`; типы `AIProvider`/`provider` удалены.
- Убран `console.log('[AI Response]', { model … })`.

**BFF: больше не проксируем тело ошибки бэкенда дословно** (`messages`, `documents/[draftId]`).

**Авторизация (BFF) — закрыты открытые роуты:**
- `POST /api/upload/presign` — требует авторизацию + владение проектом; `userId` берётся
  из сессии, не из тела; добавлен потолок размера файла.
- `GET/POST /api/projects/[id]/documents` — авторизация + владение проектом (список
  документов содержит полный OCR-текст — чувствительно).
- `DELETE /api/projects/[id]/documents/[docId]` — проверка владельца больше не
  пропускается при отсутствии `userId` (был обход).
- `POST /api/chat/[id]/messages` — проверка владения сессией (был cross-tenant inject).

**Backend: авторитетная проверка cross-tenant** (`supabase_repo.is_foreign_session` +
`chat_stream`) — работает через service-role, поэтому защищает даже когда RLS включён и
BFF не видит чужую сессию.

**Backend fail-closed** (`security.py`): при пустом `BACKEND_SHARED_SECRET` бэкенд теперь
отклоняет запросы (раньше — пропускал всех). Для локалки — `BACKEND_ALLOW_INSECURE=1`.

**Rate limiting** (`lib/rate-limit.ts` + миграция): 30 сообщений/60с и 20 OCR/5мин на
пользователя. Fail-open — сбой лимитера не роняет фичу.

**Наблюдаемость**: `user_id` добавлен в логи «History assembled» и «Chat response sent» —
теперь по логам видно, КТО отправляет запросы (текст сообщения не логируем — он в Supabase).

## ❗ Что нужно сделать руками

### 1. Применить миграции (RLS + rate limit)

```bash
supabase db push        # или примените SQL из supabase/migrations/2026070200*.sql
```

Два новых файла:
- `20260702000000_enable_rls.sql` — включает RLS и политики владения на `projects`,
  `chat_sessions`, `chat_messages`, `project_documents`.
- `20260702000100_rate_limit.sql` — таблица + функция `check_rate_limit`.

### 2. Почему RLS безопасно включать здесь

Проверено по коду, что после включения RLS приложение продолжит работать:
- **API-роуты** ходят в Supabase cookie-клиентом → выполняются от имени залогиненного
  пользователя (`auth.uid()` = он), политики `auth.uid() = user_id` пропускают его строки.
- **Python-бэкенд** ходит **service-role** → RLS обходит → история/сессии не затронуты.
- **Браузер** к таблицам обращается только через `/api/supabase-proxy`, который
  пробрасывает JWT пользователя → запросы теперь корректно ограничены его строками
  (это и закрывает дыру `rest/v1/project_documents?select=*`).

**Важный нюанс:** строки с `user_id IS NULL` (легаси/анонимные проекты и сессии) станут
невидимы для anon/authenticated роли (их увидит только service-role). Приложение сейчас
требует авторизацию, так что это допустимо. Если есть анонимные данные, которые должны
остаться видимыми, — сначала проставьте им `user_id`.

### 3. Обязательно проверить на staging перед продом

RLS с неверными политиками может «спрятать» все данные. Прогоните смоук-тест на staging
(миграции применяются к staging-проекту Supabase):
1. Логин → видны свои проекты и чаты.
2. Создать проект, создать чат, отправить сообщение — работает.
3. Загрузить документ → OCR отрабатывает; список документов виден.
4. Удалить документ — работает.
5. **Негатив:** под пользователем A попробовать `GET /api/supabase-proxy/rest/v1/
   project_documents?select=*` — должно вернуть только документы A (не всех).
6. **Негатив:** под A дёрнуть `POST /api/chat/{sessionId_пользователя_B}/messages` —
   ожидаем 404/ошибку доступа.

Только после зелёного смоука — промоут в прод (см. [[deploy-branch-workflow]]).

### 4. Прод-конфиг бэкенда
- Убедитесь, что `BACKEND_SHARED_SECRET` **задан** в проде (иначе бэкенд теперь вернёт 503).
- Идеально — снять `allow-unauthenticated-invoke` с контейнера и пускать трафик только
  через BFF (сетевой ACL/приватная сеть). Пока это так — общий секрет остаётся
  единственным забором на публичном URL.

## Возможные следующие шаги (не сделаны)
- Флаг подозрительных промптов в логах (regex «какая модель», «system prompt»,
  «ignore previous…») для алертинга по попыткам extraction.
- Выходной фильтр в стриме (regex `claude|gpt|openrouter|anthropic|openai`) как второй
  эшелон на случай, если модель всё же проговорится.
- Алерт по всплеску 429/`rate_limited` на пользователя.
