# Стейджинг на COI VM (переезд с Serverless Containers)

Одна Compute VM `jhelper-app-staging` (2 vCPU 50% / 8 GB, COI) с docker-compose:
бэкенд + фронт + Caddy (TLS) + unified-agent (логи → Cloud Logging). Деплой —
workflow **Deploy to Yandex Cloud (Staging, COI VM)**. Serverless-стейджинг
живёт параллельно и не трогается: откат = вернуться на старый URL.

Зачем VM (кратко; полное исследование — memory `yc-fargate-migration-research`):
Serverless Containers буферизуют весь ответ (SSE мёртв), морозят CPU после
ответа (висящие LangSmith-трейсы), режут всё по лимиту в 1 час (OCR больших PDF)
и вырезают `Authorization` из запроса. Ничего из этого не чинится конфигурацией.

## Разовые шаги перед первым деплоем

1. **Домен**: заведите переменную окружения `STAGING_COI_DOMAIN` в GitHub
   (environment `Staging ENV`), напр. `staging.<ваш-домен>`. Опционально
   `STAGING_COI_PROXY_URL` (аналог NEXT_PUBLIC_PROXY_URL; на VM обычно не нужен —
   same-origin, оставить пустым).
2. **SSH-ключ** (рекомендуется): секрет `STAGING_SSH_PUBLIC_KEY` — публичный
   ключ; пользователь на VM — `yc-user`. Применяется только при СОЗДАНИИ VM.
3. **Первый прогон workflow**: зарезервирует статический IP и упадёт на
   DNS-проверке, напечатав этот IP. Это ожидаемо и дёшево: VM ещё НЕ создана —
   специально, чтобы Caddy стартовал уже с рабочим DNS и получил сертификат
   Let's Encrypt с первой попытки (иначе он копит exponential backoff).
4. **DNS**: создайте A-запись `STAGING_COI_DOMAIN → <статический IP из шага 3>`.
5. **Supabase**: Auth → URL Configuration → добавить
   `https://<STAGING_COI_DOMAIN>/**` в Redirect URLs (иначе сломается
   password-recovery/magic-link flow).
6. **Повторный прогон workflow**: создаст VM, Caddy выпустит сертификат,
   смоуки должны пройти.

Права: используется тот же деплойный SA (`YC_SERVICE_ACCOUNT_ID`, роль editor)
— он же прикрепляется к VM и покрывает pull из cr.yandex и запись логов.

## Приёмка (то, ради чего переезд)

- [ ] Чат: токены печатаются **по мере генерации** (typewriter), статусы
      («Ищу судебную практику…») видны живьём, а не одним куском в конце.
- [ ] LangSmith: run закрывается сразу после ответа (нет вечных спиннеров).
- [ ] Большой PDF (>30 мин обработки) доходит до конца, а не убивается платформой.
- [ ] Auth-флоу работает (Authorization доходит; костыль x-sb-authorization
      совместим — НЕ удалять до выпила serverless).
- [ ] Логи VM видны в Cloud Logging: `resource_type=jhelper-app-staging`
      (мы явно ставим контейнерам json-file драйвер — дефолт COI `local`
      unified-agent не читает).
- [ ] Замерить пик RAM бэкенда под OCR большого PDF (`docker stats`) — это
      вход для прод-сайзинга.

## Как это устроено

- `docker-compose.tpl.yml` — шаблон compose-спеки; `render_compose.py`
  подставляет секреты/образы в CI (безопасное YAML-экранирование, `$`→`$$`,
  пустые опциональные переменные выбрасываются с логом, опечатки в токенах —
  ошибка рендера).
- Обновление: `yc compute instance update-container` — VM не пересоздаётся,
  демон COI пересоздаёт только изменённые контейнеры. Поэтому:
  - тег образа обязан меняться (используется `git rev-parse HEAD` от реально
    выкачанного кода — НЕ `github.sha`, который при re-run/откате врёт);
  - у `caddy` есть маркер-env `SITE_DOMAIN`, у `unified-agent` — `CONFIG_REV`:
    правишь heredoc-конфиг — инкрементируй/меняй маркер, иначе потребитель
    не пересоздастся и будет работать со старым конфигом.
- Откат: запустить workflow с input `branch=<старый SHA>` (checkout принимает
  SHA; теги следуют фактическому коду). Образы `staging-coi-<sha>` остаются
  в registry.
- Caddy: авто-HTTPS, `/infra/backend/*` → бэкенд (смоук из CI), остальное →
  фронт с `flush_interval -1` (мгновенная отдача SSE). Сертификаты — в
  `/home/yc-user/caddy-data` (переживают пересоздание контейнера). 443/udp
  опубликован — HTTP/3 работает, а не тихо деградирует.
- Бэкенд наружу не торчит: только compose-сеть (`backend:8080`), в SG
  открываются лишь 80/tcp, 443/tcp, 443/udp (шаг проверяет существующие
  правила и не глотает ошибки; свойство `v4-cidrs` — с `v4-cidr-blocks` yc
  падает на парсинге, этот баг жил в OpenSearch-воркфлоу).

## Известные ограничения и следующие шаги

- **Секреты в метаданных VM**: отрендеренный compose (со всеми env) виден
  через `yc compute instance get --full` любому с правом чтения инстансов и
  изнутри VM через metadata-сервис из любого контейнера. Для стейджинга —
  осознанно принято (паритет с env serverless-контейнеров, видимыми в
  консоли); для прода — секреты через Lockbox, в метаданных только ID.
- **Деплой прерывает in-flight работу** (SIGTERM→SIGKILL, ~10 с grace):
  активный SSE-стрим и идущий OCR умрут при пересоздании контейнера. Это
  сильно мягче serverless (50 мс на заморозке), но resumable-джобы для OCR
  остаются в планах. При необходимости поднять grace: `stop_grace_period`.
- **Логи → Cloud Logging**, который закрывается во II кв. 2027. Логи уже
  автоматически зеркалируются в Monium; следующий шаг — переключить output
  unified-agent `yc_logs` → `otel_logs` (агент запинен на 26.07.02 ≥ 26.04,
  уже умеет; auth через `cloud_meta`, без ключей) и поднять `CONFIG_REV`.
- **OpenSearch-VM — те же два подозрения** (вне скоупа этой ветки, чтобы не
  трогать VM с данными): шаг SG там до фикса никогда не срабатывал (порт
  открыт политикой сети), и если докер там на драйвере `local`, unified-agent
  молча не шипует логи. Проверить на VM:
  `docker inspect -f '{{.HostConfig.LogConfig.Type}}' lawers-opensearch`.
- **Лимиты памяти контейнеров** не выставлены — сначала замерить реальный пик
  бэкенда под OCR (8 GB взяты с запасом; 4 GB было бы впритык: worst-case
  ~3,0–3,4 GB суммарно).
- **Прод**: после приёмки стейджинга — тот же шаблон (VM `jhelper-app-prod`,
  100% vCPU, свой статический IP и домен, секреты через Lockbox), отдельным
  воркфлоу.
